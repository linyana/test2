# 代码审查检查清单

本清单用于在进行 Code Review 时快速核对关键点，可与 `docs/code-review-process.md` 配合使用。

## 一、整体与需求

- [ ] **需求一致性**：代码实现是否满足需求描述和业务逻辑？
- [ ] **变更范围清晰**：PR 描述是否说明了主要改动和影响范围？
- [ ] **无无关修改**：是否存在与本次需求无关的文件或格式化改动？

## 二、代码设计与结构

- [ ] **模块划分合理**：路由、服务、数据访问等职责是否清晰分层？（例如：`routes` 仅负责 HTTP/路由，业务逻辑在 `services`）
- [ ] **可读性良好**：命名是否清晰、有语义？函数和文件是否职责单一？
- [ ] **重复代码控制**：是否有可抽取为公共函数或模块的重复逻辑？
- [ ] **扩展性**：如果未来需求变更，当前设计是否易于扩展？

## 三、功能与边界情况

- [ ] **正常流程**：主流程是否覆盖并正确实现（如产品增删改查的基本场景）？
- [ ] **边界输入处理**：分页、空字符串、极大/极小值等边界输入是否处理合理？
- [ ] **错误处理**：异常和错误场景是否有清晰的返回（如 404、参数错误等）？
- [ ] **返回结果一致性**：同类接口是否保持一致的返回结构和字段命名？

## 四、类型与静态检查

- [ ] **TypeScript 类型完备**：函数参数和返回值是否合理使用类型或接口？
- [ ] **null/undefined 处理**：可能为 `null` 或 `undefined` 的值是否有显式处理？
- [ ] **静态检查通过**：是否已运行 `bun run lint` 并确认无错误（或仅存在可接受的警告）？

## 五、性能与资源

- [ ] **数据库访问合理**：是否避免了明显的 N+1 查询或不必要的多次请求？
- [ ] **分页与限制**：列表接口是否有分页或条数限制（如本项目中的 `pageSize` 上限）？
- [ ] **日志与监控**：如添加了日志，是否避免记录敏感信息和过多噪音？

## 六、安全性与健壮性

- [ ] **输入校验**：是否使用了框架/库提供的参数校验功能（如 Elysia 的 `t.Object`）？
- [ ] **错误信息**：对外暴露的错误信息是否避免泄露内部实现细节？
- [ ] **依赖使用**：是否合理使用第三方库，避免过度依赖或过时用法？

## 七、测试与验证

- [ ] **手动验证**：是否在本地通过接口/工具（如 curl、Postman、浏览器）实际验证主要用例？
- [ ] **自动化测试（如有）**：新增或修改的功能是否有对应的测试用例？
- [ ] **回归风险**：本次变更是否可能影响其他接口或模块？是否有相应验证？

## 八、文档与说明

- [ ] **文档更新**：涉及对外行为变更时，是否同步更新了 `README.md` 或相关文档？
- [ ] **提交信息**：Git 提交信息是否符合 `docs/git-commit-guidelines.md` 中的规范？
- [ ] **注释质量**：仅在必要处添加注释，注释内容是否准确、不过时？
