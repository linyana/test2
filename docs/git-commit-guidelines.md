# Git 提交规范

为了保证提交历史清晰可读、便于回溯，本仓库约定使用统一的 Git 提交信息格式（推荐采用类似 Conventional Commits 的风格）。

## 提交信息格式

基础格式：

```text
<type>(<optional-scope>): <short summary>

<optional-body>
<optional-footer>
```

- **type**：本次提交的类型（必填）
- **scope**：影响范围（可选），例如 `api`、`products`、`db`
- **short summary**：简短摘要（必填），推荐使用中文，不超过 72 字符
- **body**：详细说明（可选），用于解释动机、设计思路、注意事项等
- **footer**：额外信息（可选），例如关联 Issue、破坏性变更说明等

## type 类型约定

常用类型如下（建议从中选择）：

- `feat`：新增功能
- `fix`：缺陷修复
- `refactor`：重构（不改变对外行为的代码调整）
- `perf`：性能优化
- `docs`：仅文档修改
- `style`：代码风格调整（不影响逻辑，如格式化、空格、分号）
- `test`：测试相关变更
- `chore`：构建流程、工具、依赖更新等杂项
- `revert`：回滚之前的提交

## 示例

- 功能类：

```text
feat(products): 支持产品列表分页查询
```

- 修复类：

```text
fix(products): 修复删除不存在产品时报错信息不准确的问题
```

- 重构类：

```text
refactor(api): 抽离产品服务层逻辑以提升可测试性
```

- 文档类：

```text
docs: 补充产品 API 使用说明和示例
```

## Body 与 Footer 建议

在以下情况中，强烈建议编写 `body`：

- 设计上有多种可选方案，选择理由需要记录。
- 变更范围较大，阅读代码本身不易快速理解。
- 修改会影响到其他模块的使用方式。

在以下情况中，建议使用 `footer`：

- 关联任务或 Issue，例如：

  ```text
  Footer:
  Closes #123
  ```

- 存在破坏性变更，需要明确标注：

  ```text
  BREAKING CHANGE: 删除旧版 /products 接口，统一使用 v2 版本
  ```

## 提交前自检

在提交前，建议遵循以下步骤：

1. 确保本地代码能够编译通过。
2. 如已配置静态检查，执行 `bun run lint` 并确保无错误。
3. 检查是否包含无关变更（例如临时调试代码、无关文件等）。
4. 核对提交信息是否符合上述格式，摘要是否简洁准确。

## 与代码审查流程的关系

- 规范统一的提交信息可以帮助审查者快速理解每次变更的意图。
- 在进行 Code Review 时，审查者可以通过提交历史更容易追踪某一功能或修复的演进过程。
