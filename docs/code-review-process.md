# 代码审查流程规范

本文档用于统一本仓库的代码审查（Code Review）流程，保证代码质量、一致性和可维护性。

## 适用范围

- 所有合并到长期分支（如 `main`）的变更
- 所有功能开发分支、修复分支的 Pull Request / Merge Request

## 基本原则

- **早审查、常审查**：尽量保持 PR 体量小、提交频率高。
- **先通过自动检查，再人工审查**：本地或 CI 中通过静态检查和测试后再提审。
- **聚焦代码，而非个人**：讨论以事实和标准为依据，避免人身评价。
- **可读性优先**：在不牺牲性能的前提下，优先保证易读、易维护。

## 审查角色

- **作者（Author）**
  - 确保变更目标清晰，并在 PR 描述中说明背景、方案和影响范围。
  - 在提交前本地运行 `bun run lint`（如已配置）、必要的测试命令。
  - 及时响应审查意见，必要时补充说明或改进代码。
- **审查者（Reviewer）**
  - 按照「代码审查检查清单」逐项审查。
  - 关注设计合理性、边界情况、错误处理、性能和安全性。
  - 指出问题时提供建议性解决思路，避免只说“这样不行”。

## 审查流程

1. **创建分支**
   - 从 `main`（或约定基础分支）拉取最新代码。
   - 使用统一的分支命名，例如：
     - 功能：`feature/<短描述>`
     - 修复：`fix/<短描述>`
     - 重构：`refactor/<短描述>`

2. **开发与自检**
   - 开发过程中保持小步提交，遵循仓库的 **Git 提交规范**。
   - 开发完成后执行：
     - 静态检查：`bun run lint`（如已配置）
     - 数据库变更（如有）：确认 `prisma` 迁移脚本正确。

3. **发起代码审查**
   - 创建 PR / MR，并填写：
     - 背景与目标
     - 主要变更点
     - 风险与影响范围
     - 测试说明（例如：本地手动验证、单元测试通过等）
   - 关联相关任务或 Issue（如有）。

4. **审查与反馈**
   - 审查者按照「代码审查检查清单」进行逐项检查。
   - 建议使用「建议」「必须修改」「可讨论」等标签来区分意见优先级。
   - 对于存在分歧的问题，通过评论讨论或短会达成共识。

5. **修改与再次审查**
   - 作者根据意见进行修改，并在 PR 评论中说明关键改动。
   - 对于较大改动，可以在说明中使用列表列出修复项。

6. **通过与合并**
   - 至少 1 名有经验的开发者审批通过后方可合并（人数可根据团队约定调整）。
   - 确保：
     - CI 通过（如已配置）
     - 没有未解决的高优先级审查意见

## 代码审查检查点（概览）

详细清单见 `docs/code-review-checklist.md`，这里列出核心关注点：

- 变更是否符合业务需求，逻辑是否自洽。
- 是否遵循项目约定的 **Git 提交规范** 与编码规范。
- 接口设计是否清晰、参数与返回值是否合理。
- 错误处理是否完备，边界条件是否考虑充分。
- 是否存在明显的性能问题或资源泄露风险。
- 是否引入了不必要的复杂度。

## 文档与规范的演进

- 如在审查中发现本文档无法覆盖的场景，可在达成共识后更新本规范。
- 建议通过 PR 的方式修改此文档，以便所有人知晓规范变更。
